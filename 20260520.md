- **Senha:** *(fornecida em aula)*

> Como é Ubuntu, alguns detalhes valem destacar:

# 1. O nome resolve?

# 2. A porta 22 (SSH) está aberta a partir daqui?

## 1. Antes dos comandos: o modelo mental

Quando algo "não conecta", a pergunta certa não é *"qual comando eu rodo?"*, é *"em que camada está o problema?"*. Sempre vamos subir a pilha, de baixo pra cima:

| Camada | Pergunta | Onde investigar |
|---|---|---|
| 1. Cabo / interface | A placa de rede está viva? | `ip link` |
| 2. IP / sub-rede | Eu tenho IP? Tô na rede certa? | `ip addr` |
| 3. Roteamento | Eu sei sair da minha rede? | `ip route`, `ping gateway` |
| 4. DNS | Eu sei traduzir nome → IP? | `dig`, `/etc/resolv.conf` |
| 5. Conectividade fim-a-fim | Eu chego no destino? | `ping`, `traceroute`, `mtr` |
| 6. Porta / serviço | A porta tá aberta? O serviço responde? | `ss`, `nc`, `curl` |
| 7. Aplicação | A resposta faz sentido? | `curl -v`, logs |

**Regra de ouro:** nunca pule camadas. Se DNS não resolve, não adianta debugar HTTP.

## 2. Inspecionando interfaces e IPs

### `ip` — o comando moderno (substitui o `ifconfig`)

```bash
ip addr            # ou: ip a    → lista interfaces e seus IPs
ip link            # ou: ip l    → estado físico (UP/DOWN) das placas
ip -br addr        # versão resumida, ótima pra ler rápido
```

**O que olhar:**
- A interface está `UP`? (se estiver `DOWN`, problema de camada 1/2)
- Tem `inet 192.168.x.x/24`? Sem IP → não vai a lugar nenhum.
- O IP faz sentido pra rede em que você acha que está?

### `hostname` — quem eu sou?

```bash
hostname           # nome da máquina
hostname -I        # IPs de todas as interfaces (rápido)
```

### `ifconfig` (legado)

Ainda aparece muito em tutoriais antigos. Em distros novas pode nem vir instalado (`net-tools`). Saiba ler, prefira `ip`.

---

## 3. Roteamento: sei sair da minha rede?

```bash
ip route           # ou: ip r    → tabela de rotas
ip route get 8.8.8.8     # qual rota seria usada pra chegar nesse IP?
```

**O que procurar:**
- Uma linha `default via X.X.X.X` → esse é o **gateway**. Sem ele, você só fala com vizinhos da mesma rede.
- `ping <gateway>` antes de pingar a internet. Se nem o gateway responde, nem adianta tentar Google.

## 4. DNS: traduzindo nome em IP

A pergunta clássica: *"o site tá fora ou é só DNS?"*

### `dig` — o canivete suíço de DNS

```bash
dig google.com                    # consulta padrão (registro A)
dig +short google.com             # só o IP, sem firula
dig google.com MX                 # registros de e-mail
dig @8.8.8.8 google.com           # forçando um DNS específico (Google)
```

Se `dig @8.8.8.8 site.com` funciona mas `dig site.com` não → seu DNS local está quebrado.

### `nslookup` e `host` — alternativas

```bash
nslookup google.com
host google.com
```

Mais simples, menos detalhe. Bom pra teste rápido.

### Arquivos importantes

```bash
cat /etc/resolv.conf    # quais servidores DNS o sistema usa
cat /etc/hosts          # "DNS local" — sobrepõe DNS de verdade
```

**Pegadinha clássica:** alguém colocou uma linha em `/etc/hosts` apontando o domínio pro IP errado. O `dig` resolve certo, mas o navegador vai pro lugar errado, porque `/etc/hosts` ganha do DNS.

## 5. Conectividade fim-a-fim

### `ping` — o destino está vivo?

```bash
ping google.com           # Ctrl+C pra parar
ping -c 4 google.com      # só 4 pacotes
```

Cuidado: muitos servidores **bloqueiam ICMP** por segurança. `ping` falhar não significa que o servidor está fora — significa que não respondeu ao ping. Use também testes de porta (próxima seção).

### `traceroute` / `tracepath` — por onde meu pacote passa?

```bash
traceroute google.com
tracepath google.com      # não precisa de root
```

Mostra cada "salto" (roteador) entre você e o destino. Útil pra ver **onde** trava (ex.: passa pelo seu gateway, sai pra internet, e morre no roteador do provedor).

### `mtr` — `ping` + `traceroute` ao vivo

```bash
mtr google.com
```

Roda continuamente e mostra perda de pacote por salto. **Melhor ferramenta** pra diagnosticar instabilidade ("a internet tá lenta às vezes").

## 6. Portas e conexões abertas

### `ss` — sockets statistics (substitui o `netstat`)

```bash
ss -tuln          # TCP+UDP, listening, números (não resolve nomes)
ss -tunap         # com processo dono da porta (precisa root pra ver tudo)
ss -t state established    # conexões TCP estabelecidas
```

**Decifrando `-tuln`:**
- `t` = TCP
- `u` = UDP
- `l` = só portas em listen
- `n` = não tentar resolver nomes (mais rápido)

### Testando porta remota

A pergunta: *"a porta 443 do servidor X está acessível daqui?"*

```bash
nc -zv servidor.com 443           # netcat: -z só testa, -v verboso
curl -v telnet://servidor.com:22  # alternativa
```

Resposta típica boa: `Connection to servidor.com 443 port [tcp/https] succeeded!`

---

## 7. Testando o serviço de verdade

`ping` mostra que o IP responde. `nc` mostra que a porta abre. Mas será que o **serviço** responde direito?

```bash
curl -v https://api.exemplo.com/health
curl -I https://exemplo.com           # só os headers HTTP
curl --resolve exemplo.com:443:1.2.3.4 https://exemplo.com   # forçar IP
```

O `-v` mostra DNS, TLS, headers — tudo. É o melhor amigo do debug de API.

## 8. Firewall — será que tem alguém bloqueando?

Você não vai mexer, só **ver**. No nosso lab (Ubuntu), começa por:

```bash
sudo ufw status verbose       # firewall do Ubuntu — o principal pra gente
```

Outros que você vai encontrar mundo afora:

```bash
sudo iptables -L -n -v        # firewall clássico (por baixo do ufw na real)
sudo nft list ruleset         # nftables (moderno)
sudo firewall-cmd --list-all  # RHEL/CentOS/Fedora
```

Se você consegue conectar de outra máquina mas não dessa, **muitas vezes é firewall** (local, ou do cloud provider — Security Group na AWS, por exemplo). Foi exatamente o que aconteceu com a máquina do instrutor tentando chegar no `curso.chr.eti.br` na porta 22.

## 9. Captura de pacotes (avançado, mas vale conhecer)

Quando nada faz sentido, você abre o "raio-X":

```bash
sudo tcpdump -i any -n port 443       # tudo que passa na porta 443
sudo tcpdump -i eth0 host 8.8.8.8     # tudo de/pra esse IP
```

Não precisa dominar agora — só saber que existe. Em emergência, salva o dia.

## 10. Roteiro prático de troubleshooting

Cenário: *"não consigo acessar o site interno.empresa.com"*

```bash
# 1. Tenho rede?
ip -br addr

# 2. Tenho gateway? Ele responde?
ip route
ping -c 2 <gateway>

# 3. DNS funciona?
dig +short interno.empresa.com
# Se não resolver: dig @8.8.8.8 interno.empresa.com pra comparar

# 4. Chego até o IP?
ping -c 2 interno.empresa.com

# 5. A porta tá aberta?
nc -zv interno.empresa.com 443

# 6. O serviço responde?
curl -v https://interno.empresa.com
```

Se travar na etapa **N**, o problema está **antes ou na etapa N**. Não pule.

## 11. Resumo: o "kit de sobrevivência"

| Quero saber... | Comando |
|---|---|
| Meus IPs e interfaces | `ip -br addr` |
| Meu gateway / rotas | `ip route` |
| Se um host responde | `ping host` |
| Caminho até o host | `mtr host` |
| Resolução DNS | `dig +short host` |
| Portas abertas localmente | `ss -tuln` |
| Porta remota tá aberta? | `nc -zv host porta` |
| O serviço HTTP responde? | `curl -v url` |
| Firewall local | `sudo iptables -L -n` |
| Ver pacotes na rede | `sudo tcpdump -i any -n port X` |
